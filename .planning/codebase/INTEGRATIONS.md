# External Integrations

**Analysis Date:** 2026-03-28

## APIs & External Services

**No Third-Party Cloud Services** - Dropfall is self-contained with no external API dependencies

**Graphics Library Integration:**
- Three.js (v0.183.1) - WebGL abstraction
  - Used in `src/renderer.js`, `src/sdf/renderer.js`
  - Features: Scene, Camera, Lights, Materials, Geometries, Post-processing effects

**Physics Engine Integration:**
- @dimforge/rapier3d-compat (v0.19.3) - WASM physics simulation
  - Used in `src/physics.js`, `src/sdf/physics.js`
  - Initialization: Async WASM load via `initPhysics()`
  - Cached globally: `window.__RAPIER_WORLD__`, `window.__RAPIER_INITIALIZED__`

## Data Storage

**Databases:**
- None - Game is stateless between sessions (multiplayer state transient)

**Local Storage (Browser):**
- localStorage API for client-side persistence
  - `dropfall_settings` - Game configuration (JSON)
  - `dropfall_gamemode` - Selected game mode ('1P', '2P', 'ONLINE')
  - `dropfall_difficulty` - Difficulty level ('easy', 'normal', 'hard')
  - `dropfall_p1name`, `dropfall_p2name` - Player names
  - Accessed in `src/store.js` at startup

**File Storage (Server):**
- Node.js filesystem - Static asset serving only
  - Directory: `server/public/`
  - No persistent game data stored server-side
  - MIME type mapping in `server/server.js`

**Caching:**
- Browser HTTP cache - Static assets from Vite build
- Memory cache - Physics world object in `window.__RAPIER_WORLD__`

## Authentication & Identity

**Authentication:**
- None - No user accounts or login required
- Game fully client-side playable (single-player mode)

**Multiplayer Identity:**
- Auto-generated player IDs: `player_{incrementId}` in `server/server.js`
- Session-based only (transient, not persisted)
- No validation or security checks

**Game Identification:**
- Game IDs: `game_{incrementId}` generated by server
- Game listing: Public game query via `GET /api/games`

## Networking

**WebSocket Server (Multiplayer):**
- Library: ws (v8.18.0)
- Server: Node.js HTTP + WebSocket combined in `server/server.js`
- Port: `process.env.PORT` or default 3000
- Protocol: JSON message-based over WebSocket

**Client Connection:**
- Implementation: `src/online.js` - OnlineManager class
- Auto-protocol detection: `ws://` for HTTP, `wss://` for HTTPS
- Reconnection: Exponential backoff, max 5 attempts, 2s initial delay
- Message queue: Queues messages during disconnection, sends on reconnect

**Message Types:**
- `connected` - Player assigned ID
- `name_set` - Player name confirmed
- `error` - Error notification
- `game_update` - Game state sync
- `player_join`, `player_leave` - Lobby events
- Handled in OnlineManager.handleMessage() in `src/online.js`

**Keep-Alive:**
- Ping/pong mechanism
- `pingInterval` property in OnlineManager (interval management)

## Data Flow

**Single-Player:**
1. User input → Input handler (`src/input.js`) → Game loop → Physics step → Render update
2. Settings persisted to localStorage automatically

**Multiplayer:**
1. Client connects → WebSocket → Server assigns ID
2. Player creates/joins game → Broadcast to all players
3. Game state updates → Serialized to JSON → Sent over WebSocket
4. Server syncs all clients → All render same state

## HTTP API Endpoints

**Server REST API:**
- `GET /api/stats` - Server statistics
  - Returns: `{ totalConnections, gamesCreated, gamesCompleted }`
  - Updated every 1000ms via `broadcastInterval`

- `GET /api/games` - Available games list
  - Returns: Public games filtered by visibility

- `GET /*` - Static file serving
  - Serves: index.html, CSS, JS bundles, images, favicon
  - MIME types mapped in server configuration

## WebSocket Communication

**Connection Lifecycle (from `src/online.js`):**
1. `connect(serverUrl)` - Create WebSocket
2. `onopen` - Connection success, emit `connected` event, process message queue
3. `onmessage` - Receive JSON message, dispatch to handlers
4. `onerror` - Connection error logged
5. `onclose` - Cleanup, reset game store state
6. `send(msg)` - Queue or send JSON message

**Server Message Handlers (from `src/online.js`):**
- Message type dispatcher via `switch (msg.type)`
- Handlers update Zustand store with `useGameStore.getState()`
- State changes trigger UI re-renders

## Audio Integration

**Web Audio API (Synthesized):**
- No external audio files
- Procedural audio generation in `src/audio.js`
- Music: 135 BPM EDM tempo, speed-adjustable based on game state
- SFX: Collision sounds, power-up effects
- Gain control: `musicGain`, `sfxGain` GainNodes
- Volume: Controlled by store settings, persisted to localStorage

## Input Integration

**Keyboard:**
- Native KeyboardEvent API
- Handlers in `src/input.js`
- Configurable key bindings in store (P1/P2 controls)
- Default: WASD for P1, Arrow Keys for P2

**Gamepad API:**
- Browser Gamepad API via `navigator.getGamepads()`
- Polling in game loop
- Support for multiple gamepads (P1/P2)
- Deadzone logic for analog sticks

**Touch (Mobile):**
- Inferred from mobile detection
- Not explicitly detailed in code

## Graphics Pipeline

**Three.js Rendering (from `src/renderer.js`):**
1. Scene creation with black background
2. Camera: PerspectiveCamera (75° FOV)
3. WebGLRenderer with adaptive settings (mobile-aware)
4. Lighting: AmbientLight + DirectionalLight
5. Post-processing: EffectComposer → RenderPass → UnrealBloomPass → OutputPass
6. Shadow mapping: PCFSoftShadowMap, resolution adaptive (512x512 mobile, 1024x1024 desktop)
7. Tone mapping: ReinhardToneMapping

**SDF Alternative (Optional):**
- Shaders: `src/sdf/ray-march.glsl`, `src/sdf/sdf-functions.glsl`
- Renderer: `src/sdf/renderer.js`
- Physics: `src/sdf/physics.js`
- Game engine: `src/sdf/game-engine.js`
- Alternative rendering path (not used by default)

## Particle & Effect Systems

- `src/entities/ParticleSystem.js` - General particles
- `src/entities/LightningSystem.js` - Lightning bolts
- `src/entities/ShockwaveSystem.js` - Radial shockwaves
- All use Three.js geometries and materials

## Environment Variables
